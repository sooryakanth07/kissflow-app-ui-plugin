kf-author · Pipeline A

Big specs aren't one-shot.
They're a loop you can trust.

Upload a BRD, study what the plugin proposes, ask for changes, iterate on new suggestions — and only once you're confident about what the AI is going to build do you generate. This plugin turns that loop into commands you run in sequence.

"For any big spec, it's never one-shot generation. They upload a BRD, analyse the suggestions, offer changes, iterate — and once they're confident about what the AI is going to generate, they generate."
Worked example throughout: a Real Estate Funds Dept app — set up Funds & SPVs, process Payments & Financing, record Valuations, compute NAV, and file Fund Reports. Authored from a ~150-page spec.

Get set up (once)

Before your first run, install the plugin, stage the engine into a working folder, and give it a Kissflow access key. Five minutes, done once.

Step A · Install
/plugin
Add the marketplace and install the plugin in Claude Code. If a colleague shared the folder, point the marketplace at it directly.
Run in Claude Code
/plugin marketplace add ./kf-author-plugin     # or the shared repo/folder
/plugin install kf-app-author@kf-tools
Step B · Stage the engine
/author-setup
Run this once, in the folder you'll author from. It copies the engine, reference playbooks and seed memory into your workspace so the node engine/cli.mjs … commands resolve, and checks your prerequisites.
What it stages + checks
./engine/      ← deterministic IR → Kissflow-metadata builder + validators + tests
./reference/   ← the playbooks the agents read first
./MEMORY.md    ← auto-evolving agent memory (yours to grow)

Prerequisites checked:
  ✓ Node 18+        (zero-dependency ESM engine; `node engine/test/run.mjs` → green)
  • python3 + KF_METADATA_PATH   ← only needed for native page rendering (/author-page); skip otherwise
  • KISSFLOW_* access key         ← set this next (Step C)
Step C · Get a Kissflow key
from your DEV account
Authoring talks to the Kissflow REST API with an account-admin access key. Generate it while signed in to the environment you'll build into — your dev / builder account. must be dev
In the Kissflow web app (dev)
1. Sign in to your DEV Kissflow  →  https://dev-<company>.kissflow.com
2. Open Account Administration  →  Integrations  →  Access Keys
      (top-right avatar → "Account settings" / "Manage" → Integrations & API)
3. Create Access Key  →  copy the two values it shows:
      • Access Key ID       → KISSFLOW_API_KEY
      • Access Key Secret   → KISSFLOW_API_SECRET   (shown ONCE — copy it now)
4. Read the other two off the URL / account page:
      • Subdomain   the part before .kissflow.com   e.g. dev-acme   → KISSFLOW_SUBDOMAIN
      • Account ID  Account settings → Account details (an id like Aclo…)  → KISSFLOW_ACCOUNT_ID
Step D · Export the keys
in your shell — never commit
Export the four values in the shell you run Claude Code from. Keep them out of the repo; they live only in your environment.
Paste into your terminal
export KISSFLOW_SUBDOMAIN=dev-acme        # the subdomain — the DEV/builder one
export KISSFLOW_ACCOUNT_ID=Aclo…          # your account id
export KISSFLOW_API_KEY=<access-key-id>
export KISSFLOW_API_SECRET=<access-key-secret>
⚠ Use the DEV environment. Many Kissflow accounts have a separate dev/builder environment (dev-<company>.kissflow.com) and a runtime one — with different account ids and different access keys. A key made in one returns 403 DomainMissMatchError on the other, and artifacts only appear in the App Builder when authored with the dev key against the dev subdomain. Always generate the key in, and point KISSFLOW_SUBDOMAIN at, your dev account. The pipeline never auto-publishes to prod.

Why staged?

A Kissflow Process or Case, once published, is immutable over REST — you can't edit or delete it afterwards. So "generate first, fix later" is expensive. The pipeline moves all the thinking and iteration onto a cheap, reviewable plan, and only touches Kissflow when you're sure. Everything before /author-generate is free to change. (This app alone has 23 Processes — every one irreversible; getting them right on paper first is the whole point.)

One BRD → one run, versioned

Every BRD becomes its own run under runs/<slug>/ — isolated and versioned. Each /author-plan and /author-refine writes a new immutable snapshot (v1, v2, …) so you can compare iterations and never lose a good state.

runs/
  .current                     ← which run the commands act on
  leave-request/
    brd.md                     ← the ~150-page spec you uploaded
    app-spec.json              ← the working plan (the IR blackboard)
    decisions.md               ← every significant choice + why + rejected alternatives
    open-questions.md          ← assumptions & ambiguities to resolve
    review.html                ← the interactive review page (regenerated each iteration)
    prototype/                 ← clickable per-role UX prototype + experience-spec.json
    generated/                 ← the build log (real Kissflow ids) once you generate
    versions/
      v1/  ← snapshot: plan
      v2/  ← snapshot: after a refine
      v3/  ← snapshot: generated → dev

Stage 2 · Propose
/author-plan
Specialist agents design flow-types, data models + relationships, workflow, roles × permissions and pages — each choice logged with its rationale. Snapshots v1. plan only
Plan at a glance
Flows      Payment · Financing (Drawdown/Repayment) · Valuation · NAV · Fund Report … → PROCESSES
           Fund · SPV → FORMS      Fund Status · Currency · Payment Method → LISTS (masters)
Nesting    SPV → Authorized Agents, POA List   ·   Fund → Project Location   (child tables)
Roles      Fund Manager · Compliance · Finance · Operations · BOD Member · Admin
Workflow   Payment:  Create Payment → Manager Approval → Finance Processing → Paid
Computed   Payment.Total = Σ line items (aggregate)   ·   Fund.Remaining = Financing − Drawdown (formula)
Counts     23 processes · 26 lists · 20 child tables · 72 permission grants

Top decisions (decisions.md):
  • Payment / Financing = Processes — each has an approval lifecycle, not a static record.
  • SPV stays a Form (no lifecycle) but is child-tabled under Fund for the one-to-many.
  • NAV is computed from the latest Valuation — a derived field, never entered.

The two things reviewers actually look at

Teams understand data models & relationships, business logic, workflows, and pages/widgets. Review gives you two complementary views of the same plan:

1 · review.html

The interactive review page

A structured, self-contained HTML document — the same template for every use case. Every entity, field, workflow, permission, page and decision is an item with a stable #id. Mark it ✓ ok / ✎ change / ? ask, add a comment, then hit Copy change-list to get a paste-ready list of [CHANGE] <label> (#id): note lines. Feed that straight into /author-refine. Flags persist in the browser, so the whole team reviews one file.

2 · prototype/

The clickable prototype

A separate, constraint-free UI agent (kf-prototype) builds a per-role experience — nav + pages + widgets — from the outcomes each role needs, with realistic mock data. Here the Fund Manager lands on a "Fund Overview" with AUM & pending-payment KPIs, Finance on a "Payments Queue". It deliberately ignores Kissflow artifact limits, so reviewers judge the experience, not JSON.

The prototype isn't throwaway — it becomes the live UI

The prototype is built to a Kissflow-agnostic Experience Spec (prototype/experience-spec.json) whose widgets bind to your IR entities by meaning (entity / field / measure / dimension / scope), not to Kissflow ids. Because Pipeline B (kf-framework) is also a flexible React runtime, the same spec becomes the live custom UI after generation — nav, per-role pages, widget types and layout translate 1:1; only the data source swaps from mock to real.

Prototype (A)

"Fund Overview" AUM KPI + fund table rendered with mock data — for review.

Experience Spec

kpi bind:{entity:"Fund", measure:"sum", field:"Total Fund size", scope:"all"} → "AUM".

Live UI (B)

Same KPI, bound to the real Fund form's field — production React.

The express path — demos & small specs

You can skip review and generate directly. The input can be a BRD file, pasted requirement text, or even a one-line ask — a file isn't required:

/author-app "./specs/leave-request.md"                       # a BRD file
/author-app "a purchase request app with two-level approval and a vendor master"   # one-liner
/author-app "<paste a paragraph of requirements here>"  # pasted text
                                      # --dry-run stops at the manifest · --yes skips the confirm

The thinner the input, the more the plugin leans on assumptions + open-questions — a one-liner is valid; you just review more of what it inferred. The same goes for /author-brief in the staged flow.

⚠ Heads-up. The express path prints No review taken — applying to dev directly. and builds in dev. /author-generate is gated on a valid IR, not on review, so this is fully supported — but for a 23-process app like this one, use the staged loop. The run is still saved and versioned, so you can /author-refine and regenerate afterwards.
