Kissflow App Metadata

The complete structure of a low-code app — and how to author it through the API.

Verified against kissflow-xg source Live structures from a real Kissflow app export For the kf-app-author plugin

Overview & how to read this

A Kissflow application is a tree of metadata objects. Some are simple documents (the app, its variables, a role); the two hard ones — a model (form/process/case) and a page — are stored as a normalized entity graph: a flat dictionary keyed by entity Id, with a Root pointer, where every parent references its children by id arrays named Parent::ChildType (e.g. Model::Field, Field::Expression).

Model / Flow Layout Field Reference / Query Expression / Node Page Role / Permission
Throughout, {acct} = account id, {appId} = application id, {type} ∈ form | process | case | dataset. Authoring writes go to the /flow/2 and /metadata/2 surfaces; runtime records are a different API.
Complete attribute catalogs. The structures shown here are the common shape. The exhaustive, source-cited attribute lists — every .get() key, every relation, every enum value, with file:line from the kissflow-xg source — live in METADATA_ATTRIBUTES_model.md (Field has ~26 scalar attrs + 21 :: relations; Model ~22 + 13 relations) and METADATA_ATTRIBUTES_page.md (page/view/workflow/permission/app). Treat those as authoritative; this page is the orientation.

Concepts & context — what the metadata means

The structures below say how metadata is shaped; this says what it means to an app builder. A Kissflow App bundles three things — data, workflow, and UI — into one interconnected unit. (Product semantics from the Kissflow community Apps & Portals docs.)

The mental model: a page hosts components that bind to a process/form/case/dataset/report; a dataset/list feeds the reference/select fields of forms; a case is a board whose statuses are columns; a process routes a record through role-owned steps; app variables carry computed numbers into KPI cards; navigation + roles wire and gate it all.
Why it matters for authoring: build along the product grain — masters/lists → forms → processes/cases that reference them → views → roles → pages. A blob that ignores these links (a Reference with no target, a card bound to no report, a status with no transition) parses but is a broken app. Build meaning, not just shape. See CONCEPTS.md.

The flat-dictionary format means there is no "create field" endpoint. You add a field by inserting its entity (plus a Column to host it and its id into Model::Field) and re-publishing the whole blob.

Layout — rows, columns, sections

The form's visual structure is a Row/Column tree. A Column is either a Section (a titled group that holds more rows) or a Field slot. Start/End place the column on a 0–6 grid (so 0–3 + 3–6 = two half-width fields side by side).

// two fields on one row, each half width
"Row_fCJZKJSCsL": { "Kind":"Row", "Column":"Column_Cne5A1mxDU",
                   "Row::Column":["Column_Q5ZTV-Mv8t", "Column_VAsYqssg5"] },
"Column_Q5ZTV-Mv8t": { "Type":"Field", "Start":0, "End":3, "Column::Field":["Asset_Tag_Id"] },
"Column_VAsYqssg5": { "Type":"Field", "Start":3, "End":6, "Column::Field":["Asset_Serial_Number"] }

"IsHidden": true on a column hides a field (common for computed helper fields that feed expressions but aren't shown).

References & query definitions

A Reference (or User) field looks up rows from another model. The lookup is a QueryDefinition: the target model (LHSModel), which columns to pull (LookupField), which to hide, and an optional filter (Criteria).

"Asset_Category": { "Kind":"Field", "Type":"Reference", "Name":"Asset Category",
                    "Field::QueryDefinition":["QueryDefinition_Jow8S9MO9"], "Required":true },
"QueryDefinition_Jow8S9MO9": {
  "Kind":"QueryDefinition", "Field":"Asset_Category",
  "LHSModel":"Asset_Categories", "FlowType":"Form",
  "LookupField":[{ "Id":"Category_Id", "Name":"Category Id", "Type":"Text" },
                 { "Id":"CategoryName", "Name":"Category Name", "Type":"Text" }],
  "HiddenField":["Category_Id"] }
This is why authoring order matters: a model with Reference fields must be created after the model it points to (the LHSModel must exist).

Criteria & conditions (filters)

A QueryDefinition (or a validation, or visibility rule) can carry a Criteria → Condition[]. A condition compares a left field to a value or another field.

"Criteria_BT-HAFTn8": { "Kind":"Criteria", "QueryDefinition":"QueryDefinition_jQDSFse8N",
                       "Criteria::Condition":["Condition_gcOZNOR8xI"] },
"Condition_gcOZNOR8xI": { "Kind":"Condition", "LHSField":"Asset_Master_Id",
                        "Operator":"EQUAL_TO", "HasArguments":true,
                        "RHSType":"Field", "RHSField":"Asset_Master_Id" }
// RHSType:"Value" → use RHSValue:"In Store" instead of RHSField

RHSType ∈ Value · Field · FilterParam. Operator — 25 in source, incl. EQUAL_TO, NOT_EQUAL_TO, GREATER_THAN, GREATER_THAN_OR_EQUAL_TO, LESS_THAN, LESS_THAN_OR_EQUAL_TO, CONTAINS, NOT_CONTAINS, STARTS_WITH, ENDS_WITH, IS_EMPTY, IS_NOT_EMPTY, IN, NOT_IN, BETWEEN … (full list in METADATA_ATTRIBUTES_model.md).

The same Criteria → Condition shape powers four things, via different relation arrays on the field: lookup filters (QueryDefinition::Criteria), validations (FieldValidation::Criteria), visibility (ColumnVisibility::Criteria), and conditional-required (FieldRequirement::Criteria). Condition subclasses: TableAssignment · Validation · Visibility · Requirement · Query · Grid.

Expressions & node trees (computed fields)

A computed field carries a Field::Expression. The Expression stores both the human formula (ExpressionStr) and a parsed Node tree — an ordered graph the engine executes. Node Type ∈ Function · Static · Field · RefField; chained calls carry Syntax:"Chain".

Example 1 — a reference lookup: Asset_Category.CategoryName

"Expression_Kr6Og2ERr": { "Kind":"Expression", "Field":"Category_Name",
   "ExpressionStr":"Asset_Category.CategoryName", "Expression::Node":["Node_EoBvUw6A9o"] },
"Node_EoBvUw6A9o": { "Type":"Function", "Value":"GetValue", "Category":"Reference",
                   "DataType":"String", "Node::Node":["Node_NiqHEF987a", "Node_2r8jRlLIlj"] },
"Node_NiqHEF987a": { "Type":"Field",    "Field":"Asset_Category", "DataType":"Reference" },
"Node_2r8jRlLIlj": { "Type":"RefField", "RefField":"CategoryName", "RefModel":"Asset_Categories" }

Example 2 — functions: now() and a chained concatenate(...)

// now() → one Function node
"Node_xYzoV4shc2": { "Type":"Function", "Value":"now", "Category":"DateTime", "DataType":"DateTime" }

// concatenate("ACO", initiatedat().day().toText(), ...) → chained functions
"Node_OLXmaQhnvad": { "Type":"Function", "Value":"concatenate", "Node::Node":["Node_X3..", "Node_HV..", ...] },
"Node_X3BxlL-oeWk": { "Type":"Static", "Value":"ACO", "DataType":"String" },
"Node_HVej5jTprIt": { "Type":"Function", "Value":"toText", "Syntax":"Chain", "Node::Node":["Node_sV.."] }
// → ...day().Chain → initiatedat()  (chains read inner-most first)

Node types & attributes (complete)

The engine's real enum is NodeType = Function · Static · Field · AggregateFunction · AggregateField · FetchField (live data also carries RefField). A Node may hold:

Type  Value  Field  DataType  Category  RefModel  RefField  RefRootModel
LookUpPath  FunctionFlowType  Expression  FieldRefCount  Syntax  + Node::Node
Not enums (important for a generator): DataType is a flat list (String/Number/DateTime/Reference/Boolean…), Category is the function registry's namespace keys, and Syntax:"Chain" has no enum constant in source — it's a free-form string. Nothing validates these server-side, so they must be written exactly right.
Authoring a formula means writing both ExpressionStr and the matching node tree. The simplest robust path: use the access-key AI flow_builder or the fields/suggest endpoint to generate the node tree from the formula string.

Workflow & caseflow

Steps and statuses are not separate endpoints — they live inside the process/case draft blob and publish with it.

Process — Workflow.Step[]

Each step is an Activity: a name, a type, an assignee (Resource{ValueType:"AppRole", Value:<roleId>}), and per-step field permissions (Activity::Permission).

Author via PUT /metadata/2/{acct}/process/{id}/draft → publish.

Case — caseflow

A status machine stored as its own blob (Kind:"CaseFlow") with CaseFlow::Status and CaseFlow::State. Transitions are inline: each Status lists its allowed next statuses in OutwardStatus[].

"Status_0001_New": { "Kind":"Status", "Name":"New",
   "Category":"NotStarted", "IsSystem":false,
   "OutwardStatus":["Status_InProgress", ...],
   "EntryRule":[], "ExitRule":[], "Rule":[],
   "SLADisabled":false, "SLABreachResources":[], "ColorCode":"" }
"State_Not_Started": { "Kind":"State", "Category":"NotStarted",
   "IsDefaultState":true, "IsLastState":false }

Status.Category ∈ NotStarted · InProgress · Done · Closed · ReOpened. A parallel casepermission blob holds Permission{Status, Column, Permission}.

Roles & permissions

Two layers. Roles (app_role) are simple docs with members — and creating them is access-key reachable (verified live). Permissions are discrete objects inside the relevant draft blob, referencing the role by _id.

POST /app_role/2/{acct}/
{ "Name":"Admin Head", "Description":"...", "_application_id":"...",
  "Users":[{ "_id":"Us..." }], "Groups":[] }   // → { "_id": "RoCb_..." }

POST /app_role/2/{acct}/application/{appId}/external/batch   // grant app access
[{ "_id":"RoCb_...", "_application_id":"{appId}" }]
Permission lives in…Shape
Form viewModel::Permission → Permission{Column, Permission: Editable|ReadOnly|Hidden}
Process stepActivity::Permission (+ Mandatory), assignee via Activity::Resource
Casecasepermission → Permission{Status, Column, Permission}

Permission levels (field visibility): Editable · ReadOnly · Hidden (process steps add Mandatory). Capability/action permissions are a large enum in base/base/auth/permissions.py — ~76 Case, ~22 Process, 16 Project, plus Form/Dataset/Application/Team/DecisionTable sets — bundled into built-in roles by the *RoleDefinition dicts in roles.py.

A process step's assignee is a Resource{ValueType, Value} where ValueType ∈ User · Group · AppRole · ServiceAccount · Field · Expression · FlowDataset · FlowForm · FlowUser · FlowTable. Workflow step kinds: StartEvent · UserTask · MultiUserTask · Parallel · GotoTask · SendBackToInitiator · Subflow · EndEvent.

Field-level permissions are positional, not keyed by role name — the role↔field binding is by the role's _id in the permission object.

Navigation

The app's menu is its own blob: Navigation → Menu → SubMenu → Property (distinct from pages).

Authoring API — what an access key can do

Derived from the kissflow-xg source. Public /2/ = reachable; internal /kfv2/ = service-only (not reachable).

ActionEndpointAccess key?
Create appPOST /flow/2/{acct}/application/public
Create model shellPOST /flow/2/{acct}/{type}/yes ✓
Author fields / layout / stepsPUT /metadata/2/.../{type}/{id}/draft → publishX-Api-Key class
AI generate a draft from a promptPOST /metadata/2/{acct}/flow_builder/{type}/createyes ✓

Creating a model/field has no direct public endpoint — you mint a shell (/flow/2/.../{type}/, which triggers the internal bootstrap for you) and then write the body through draft/publish. A blind POST /form/2/... 404s.

Authoring order

Build an app in dependency order so references resolve:

1. Application # POST /flow/2/{acct}/application/ 2. Lists / Datasets / masters # so Reference fields have a target (LHSModel) 3. Forms # create shell → draft fields/layout → publish 4. Processes / Cases # may reference the forms above; add workflow/caseflow 5. Views # formview/caseview draft → softpublish 6. Roles # app_role create + grant app access (access-key ✓) 7. Permissions # field/step/status perms keyed by role _id 8. Pages # native page graph — OR hand to kf-app-builder for React UI
